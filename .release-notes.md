v0.3.0: reshape for lazysite D013 architecture

Bundle of six LL-SMs: directory reshape, packaging script
rewrite, docs rewrite, theme.json D013 schema + CSS-variable
wiring, manager/ removal, first release under the new layout.
Aligned with lazysite 0.2.10's D013 architecture. Pre-D013
lazysite versions cannot consume the new zips.

### LL-SM001 - directory reshape

Flattened and re-nested:

    dark/                       # removed
    default/                    # removed (one view.tt retained as layout.tt)
    manager/                    # removed entirely (see LL-SM005)

becomes:

    layouts/default/
      layout.tt                 # from default/view.tt, inline <style> stripped
      layout.json               # new metadata file
      themes/
        default/
          theme.json            # D013 schema with config tokens
          assets/main.css       # tokenised with var(--theme-*)
        dark/
          theme.json
          assets/main.css       # identical to default's main.css

The inline `<style>` block that used to live at the top of
`default/view.tt` (colours and typography baked in) has been
extracted into `assets/main.css`. Every colour reference in
`main.css` now points at a CSS custom property via `var()`.
The same `main.css` ships for both themes - only the
`theme.json.config` values differ, which the processor emits
as `:root { --theme-GROUP-KEY: ...; }` via `[% theme_css %]`.

### LL-SM002 - tools/package-themes.sh rewrite

New discovery walk: `layouts/LAYOUT/themes/THEME/`. Each
theme dir produces `releases/LAYOUT/THEME.zip` (layout-nested;
mirrors source structure so future layouts shipping a
"default" theme don't collide). The zip contents follow the
D013 upload shape:

    theme.json         # at zip root
    assets/main.css    # web-accessible via /lazysite-assets/LAYOUT/THEME/
    assets/...         # any other theme-shipped files

Notable differences from the old script:

- Walks two levels deep from `layouts/`, not top-level
  directories.
- Requires `theme.json` at each theme dir; warns if
  `assets/main.css` is missing.
- Never includes `layout.tt`, `layout.json`, or `nav.conf` in
  a theme zip. Layouts are distributed by other means; nav is
  site content.
- Clears `releases/` before rebuilding so stale zips don't
  persist when a theme is renamed or removed.
- `zip` is primary; falls back to `python3` if `zip` is
  missing (portable across minimal containers).
- Uses `$$` for temp-dir paths per CLAUDE.md (no `mktemp`).

Verified outputs:

    $ bash tools/package-themes.sh
    Packaging: default/dark
      Built: releases/default/dark.zip (3716 bytes)
    Packaging: default/default
      Built: releases/default/default.zip (3721 bytes)
    Done: 2 theme(s) packaged at 7K total

    $ python3 -c "import zipfile; print(zipfile.ZipFile('releases/default/default.zip').namelist())"
    ['assets/', 'assets/main.css', 'theme.json']

Both zips pass the D013 shape check.

### LL-SM004 - theme.json D013 schema + config wiring

Both `default/theme.json` and `dark/theme.json` rewritten to
the D013 schema:

- Required: `name`, `version`, `description`, `author`,
  `layouts` (non-empty array), `config` (grouped tokens
  object).
- `layouts`: `["default"]` for both.
- `config`: thirteen tokens each under `colours` (11 entries)
  and `fonts` (2 entries). Every token corresponds to a
  `var(--theme-GROUP-KEY)` reference in `main.css`:

      colours.primary             -> --theme-colours-primary
      colours.primary-hover       -> --theme-colours-primary-hover
      colours.text                -> --theme-colours-text
      colours.text-muted          -> --theme-colours-text-muted
      colours.text-dim            -> --theme-colours-text-dim
      colours.heading             -> --theme-colours-heading
      colours.background          -> --theme-colours-background
      colours.background-alt      -> --theme-colours-background-alt
      colours.background-accent   -> --theme-colours-background-accent
      colours.background-dropdown -> --theme-colours-background-dropdown
      colours.border              -> --theme-colours-border
      fonts.body                  -> --theme-fonts-body
      fonts.code                  -> --theme-fonts-code

- Palette values:

      default: light, #0056b3 primary, #fafafa background
      dark:    dark,  #6db3f2 primary, #1a1a1a background

- `files` array lists `theme.json` and `assets/main.css`.
  Informational only (packager walks the directory).

The version bump (old files were `1.0.0` but with pre-D013
schema) stays at `1.0.0` - the filing under the new repo
name at tag `v0.3.0` is the breaking-change marker, not a
per-theme version bump.

### LL-SM005 - manager/ removed

The old `manager/` directory held an orphaned copy of the
lazysite manager UI template (57-line `view.tt` emitting
`<body class="mg-body">` and navigation into `/manager/*`),
not a user-facing site theme. The audit confirmed this file
is now redundant: lazysite 0.2.10's D013 work ships the
manager UI internally at `{DOCROOT}/lazysite/manager/layout.tt`,
so the copy in this repo had no consumer.

Action: deleted entirely. No slate theme, no rename. A
genuine slate-aesthetic theme is design work and belongs in
a future SM (call it D014).

Net result: this release ships two themes (default, dark),
down from three (default, dark, manager).

### LL-SM006 - first release under new structure

Artifacts (layout-nested):

- `releases/default/default.zip` - light theme (3,721 bytes)
- `releases/default/dark.zip` - dark theme (3,716 bytes)
- `releases/default.zip`, `releases/dark.zip`,
  `releases/manager.zip` - old flat paths, all removed

Path scheme: `releases/LAYOUT/THEME.zip`. Layout-nested so a
future second layout (e.g. `studio`) can ship its own
`default` theme without a filename collision.

Both zips smoke-tested: `theme.json` at root, `assets/main.css`
at `assets/`, no `layout.tt`, no `nav.conf`, no `assets/`
wrapper at root (the wrapper is under `assets/` which is
correct per D013). `theme.json` has all six required D013
fields including `layouts[]` and `config{}`.

### Scope corrections from audit

Three briefing errors caught and resolved during the audit
(decisions shipped in the accepted scope):

- **manager/ is not a user theme.** Briefing treated it as one
  and asked for a rename to "slate". The file is actually the
  lazysite manager UI chrome, already shipped internally by
  lazysite core. Deleted instead of renamed.
- **Zip shape.** Briefing said "theme.json + main.css at root,
  no assets/ wrapper". D013's installer only web-serves the
  `assets/` subtree. The correct zip shape is `theme.json` at
  root + `assets/main.css` (not root main.css). Packager
  updated accordingly.
- **Baked-in CSS.** Briefing's scope guardrail said "don't
  rewrite layout.tt beyond D013 compatibility". But the old
  `view.tt` had ~400 lines of inline `<style>` with colours
  hard-coded, so D013's theme_css would've been decorative
  only. Lifted the guardrail on audit sign-off; inline CSS
  extracted to per-theme `main.css` with `var()` substitutions
  so `theme.json` actually drives rendering.

### Compatibility

Target: lazysite **0.2.10+**. Earlier versions don't recognise
D013's `theme.json` schema (no `layouts[]`, no `config{}`) and
the manager would reject any upload from this repo's zips.

No migration path for operators on older lazysite. They
upgrade lazysite first, then install a theme from here.
